Nothing reachable from the open internet

Every request clears independently-scoped layers. No single misconfiguration exposes a service — a failure that opens one plane is caught by the next.

Ingress path

client device │ per-device WireGuard key (Tailscale mesh) ▼ Tailscale Serve — HTTPS on the tailnet iface only │ proxies to services bound to 127.0.0.1 ▼ UFW host firewall — each subnet allow-listed per port │ ▼ segmented Docker networks — isolated bridges

What that buys

  • VPN-only ingress. Per-device key auth is the sole way in; no ports forwarded on the router.
  • Loopback binding. Services listen on 127.0.0.1, exposed only via the tailnet-facing proxy.
  • Firewall as an independent second layer. The VPN failing open still would not expose services.
  • Network segmentation. Isolated bridge networks limit lateral movement between services.

Security posture

Independently-scoped controls, hardened by default — with one trade-off documented on purpose.

Zero public attack surface

Every service sits behind the mesh VPN; nothing is reachable from the open internet.

Defense in depth

The firewall layer is scoped independently of the VPN, so a misconfigured container can't widen its audience.

Least privilege

cap_drop: ALL, narrow cap_add, no-new-privileges everywhere, non-root UID/GID and read-only rootfs where supported.

Secrets out of configs & VCS

Credentials injected via environment files or mounted read-only secret files — e.g. scrape jobs authenticate via bearer_token_file, never a token in a checked-in config.

Supply-chain-aware automation

Automatic image updates aren't applied blindly: every stateful, data-bearing service is excluded from auto-following tags, so patch automation can't silently corrupt a database.

One documented, conscious exception

Known trade-off

cAdvisor still runs in Docker's privileged mode rather than a minimal capability set. This was evaluated and deliberately deferred, not overlooked — cAdvisor needs deep host/cgroup introspection, and a wrong minimal-capability set tends to fail silently (metrics quietly stop reporting instead of erroring), which is a worse outcome than the known, contained trade-off. Hardening blindly here would swap a visible risk for an invisible one.

Incidents & lessons

Kept because the debugging process is more informative than a clean success story.

CORRUPTIONINC-01

An auto-updater corrupted a brand-new deployment

Standing up Paperless-ngx, the update tracker — watching Docker's live event stream — grabbed the Postgres and Redis containers within seconds of creation and pushed them to tags it misjudged as "newer": a pre-release DB beta and a legacy-arch cache build.

New stateful services now get an explicit auto-update opt-out label before first deploy.

HTTP 400INC-02

One 400 error was three unrelated bugs stacked

Peeled back in order: (a) the app didn't trust its proxy's forwarded headers; (b) that exposed a silent Bluetooth retry loop blocking clean restarts; (c) a real port collision between the VPN listener and the app's wildcard bind. Each fix revealed the next.

Don't declare victory after the first plausible-looking fix.

TIMEOUTINC-03

Metrics scraping failed with no app error at all

Scrapes timed out intermittently while app config, DNS and container health looked fine. The cause was the host firewall: one Docker subnet was allow-listed to reach the port, but a second subnet that needed it was missed.

"Works from A but not B" almost always means the network/firewall layer.

Roadmap

A build worth trusting names what it hasn't solved yet. These are the known gaps, in priority order.

Backups

Biggest gap

Nothing is backed up yet — password vault, DNS config, document store, log history, photo library. The plan is Kopia against a NAS once it's online, deliberately not a sleep-prone laptop, which is an unreliable always-on backup target in practice.

Storage migration

In progress

Moving the root filesystem and volumes off the SD card onto NVMe (WD Black SN770), removing SD-card wear and I/O contention as a reliability bottleneck.

Home Assistant

In progress

Onboarding is done; still connecting the first smart device and finishing API integration into the monitoring stack.
