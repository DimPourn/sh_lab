01 / 06 · defense in depth
01 — the edge

The internet stops here.

No ports are forwarded on the router and nothing binds to a public interface. There is no listening service for a scanner or bot to find — the usual first move of an attack, "find an open port", has nothing to land on.

router: 0 inbound forwards
nmap from the WAN: no open ports
attack surface: none exposed
02 — wireguard mesh

The only way in.

Tailscale builds a private WireGuard mesh where every device authenticates with its own key — no shared password, no public endpoint. Access is identity-based per device: revoking one key cuts that device off instantly without touching anything else, and traffic is end-to-end encrypted.

tailscale · per-device keys
ingress: VPN-only
revocable, identity-based access
03 — tailscale serve

TLS on the tailnet.

Tailscale Serve terminates HTTPS on the tailnet interface and reverse-proxies to services bound to 127.0.0.1 — nothing listens on a public-facing wildcard address. Certificates for the tailnet hostname are issued and renewed automatically, so every service is reachable over HTTPS without exposing a port or hand-managing certs.

https → 127.0.0.1
no wildcard binds
auto-issued & renewed TLS
04 — ufw host firewall

An independent second layer.

UFW allow-lists each Docker subnet to only the specific ports it needs, scoped independently of the VPN. Every rule names a source subnet and a destination port, so a service can only be reached by the containers that legitimately need it — e.g. only the monitoring subnet may hit an app's metrics port. If the VPN ever failed open, services still would not be exposed.

per-subnet allow-lists
least-privilege network paths
defense in depth, not "the VPN handles it"
05 — segmented networks

No flat wire.

Services are split across isolated Docker bridge networks — a monitoring net, a reverse-proxy net — instead of one flat network where everything can talk to everything. A container only joins the networks it actually needs, so a compromise stays boxed inside its own segment with no lateral path to unrelated services.

isolated bridges
no lateral movement by default
contained blast radius
06 — the hardened core

Least privilege.

Every container starts from zero Linux capabilities — cap_drop: ALL — and gets back only the few it genuinely needs. no-new-privileges blocks escalation, processes run as an unprivileged user, and root filesystems are read-only where the image supports it. Even a full app compromise stays boxed in a low-privilege, mostly read-only container.

02 — how it's hardened

Security posture

Independently-scoped controls, hardened by default — with one trade-off documented on purpose.

Zero public attack surface

Every service sits behind the mesh VPN; nothing is reachable from the open internet.

Defense in depth

The firewall layer is scoped independently of the VPN, so a misconfigured container can't widen its audience.

Least privilege

cap_drop: ALL, narrow cap_add, no-new-privileges, non-root UID/GID and read-only rootfs where supported.

Secrets out of configs & version control

Credentials injected via environment files or mounted read-only secret files — e.g. scrape jobs authenticate via bearer_token_file, never a token in a checked-in config.

Supply-chain-aware

Every stateful, data-bearing service is excluded from auto-following image tags, so patch automation can't silently corrupt a database.

One documented, conscious exception known trade-off

cAdvisor still runs in Docker's privileged mode rather than a minimal capability set — deliberately deferred, not overlooked. It needs deep host/cgroup introspection, and a wrong minimal-capability set tends to fail silently (metrics quietly stop reporting instead of erroring), a worse outcome than the known, contained trade-off.

Network segmentation

Multiple isolated Docker networks instead of one flat network, to contain the blast radius of any one container.

03 — real failures, fixed

Incidents & lessons

Kept because the debugging process is more informative than a clean success story.

CORRUPTIONINC-01

An auto-updater corrupted a brand-new deployment

Standing up Paperless-ngx, the update tracker — watching Docker's live event stream — grabbed the Postgres and Redis containers within seconds of creation and pushed them to tags it misjudged as "newer": a pre-release DB beta and a legacy-arch cache build.

New stateful services now get an explicit auto-update opt-out label before first deploy.

HTTP 400INC-02

One 400 error was three unrelated bugs stacked

Peeled back in order: (a) the app didn't trust its proxy's forwarded headers; (b) that exposed a silent Bluetooth retry loop blocking clean restarts; (c) a real port collision between the VPN listener and the app's wildcard bind. Each fix revealed the next.

Don't declare victory after the first plausible-looking fix.

TIMEOUTINC-03

Metrics scraping failed with no app error at all

Scrapes timed out intermittently while app config, DNS and container health looked fine. The cause was the host firewall: one Docker subnet was allow-listed to reach the port, but a second subnet that needed it was missed.

"Works from A but not B" almost always means the network/firewall layer.
