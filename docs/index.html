[01] Network & access model

Nothing on this box is reachable from the open internet. Access flows through layers that are each scoped independently, so no single misconfiguration exposes a service:

client device │ per-device WireGuard key (Tailscale mesh — no router port forwarding) ▼ Tailscale Serve — HTTPS termination on the tailnet interface only │ reverse-proxies to services bound to 127.0.0.1 ▼ UFW host firewall — each Docker subnet allow-listed per port, independently of the VPN │ ▼ segmented Docker bridge networks — monitoring / proxy-facing / app networks kept isolated
  • VPN-only ingress. A WireGuard mesh with per-device key authentication is the sole way in; no ports are forwarded on the router.
  • Loopback binding. Services listen on 127.0.0.1 and are only exposed via the tailnet-facing reverse proxy — nothing binds a public wildcard address.
  • Firewall as a second, independent layer. Each internal Docker subnet is explicitly allow-listed for only the specific ports it needs. The VPN failing open would still not expose services.
  • Network segmentation. Services are split across isolated bridge networks instead of one flat network, limiting lateral movement between unrelated services.

// segmentation

Isolated networks, not one flat wire

Services are split across separate Docker bridge networks with a UFW layer on top — so a single compromised container can't roam the rest.

[03] Security posture

  • Zero public attack surface — every service sits behind the mesh VPN; nothing is reachable from the open internet.
  • Defense in depth — the firewall layer is scoped independently of the VPN, so a misconfigured container can't accidentally widen its audience.
  • Least privilege on containers — cap_drop: ALL with narrow, explicit cap_add only where genuinely needed; no-new-privileges everywhere; non-root UID/GID mapping and read-only root filesystems where the image supports it.
  • Secrets out of configs and version control — credentials injected via environment files or mounted read-only secret files (e.g. scrape jobs authenticate via bearer_token_file, never a token in a checked-in config).
  • Supply-chain-aware automation — automatic image updates are useful but not blind: every stateful, data-bearing service is excluded from auto-following tags, so patch automation can't silently corrupt a database.
// under the hood

Where the interesting failures live

Every layer that can break has broken at least once. The debugging is documented below — it's the part worth reading.

[04] Incidents & lessons learned

Real problems hit and fixed during the build — kept because the debugging process is more informative than a clean success story.

INC-01An auto-updater silently corrupted a brand-new deployment

While standing up Paperless-ngx, the update tracker — which watches Docker's live event stream, not just a schedule — grabbed the Postgres and Redis containers within seconds of creation and pushed them to tags it misjudged as "newer": a pre-release database beta and a legacy-architecture cache build. Root cause: tag comparators can't reliably tell a stable release from a beta/RC/legacy tag across every image's own versioning convention.

Any new stateful service gets an explicit auto-update opt-out label before its first deploy, not after.

INC-02A reverse-proxied service returning 400s was three unrelated bugs stacked together

Diagnosis peeled back in order: (a) the app didn't trust its reverse proxy's forwarded headers; (b) fixing that exposed a silent Bluetooth-integration retry loop quietly preventing the web server from restarting, with no visible crash; (c) a genuine port collision between the VPN's own listener and the app's wildcard bind. Each fix revealed the next problem.

Don't declare victory after the first plausible-looking fix.

INC-03Cross-service metrics scraping failed with no application-level error at all

Scrape requests timed out intermittently while app config, DNS, and container health all looked fine. The actual cause was the host firewall: one Docker subnet had been allow-listed to reach the target port, but a second subnet that needed the same access was simply missed when the rule was written.

"Works from container A but not container B" almost always means the network/firewall layer, not the app.
